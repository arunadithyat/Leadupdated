SECTION 2: IMMEDIATE FIXES REQUIRED (HIGH SEVERITY)
================================================================================

⚠️ CRITICAL BUG #1: callStarted Flag Set to False After Auto Call
    Location: lib/main.dart, line ~1265
    File: lib/main.dart
    
    PROBLEM:
    When initiating a call, the code does:
        final success = await AutoDialer.autoCall(mobileNo);
        callStarted = false;  // ← WRONG!
    
    The flag is ALWAYS set to false, even if call succeeded.
    Should be: callStarted = success;
    
    IMPACT:
    - Call state not tracked properly
    - Completion dialog never shows
    - Call duration tracking doesn't start
    - Call status not logged
    
    FIX: Change line to read:
        callStarted = success;
    
    SEVERITY: HIGH (Blocks core functionality)
    EFFORT: 5 minutes

---

⚠️ CRITICAL BUG #2: No Input Validation on Phone Numbers
    Location: lib/services/auto_dialer.dart, lib/api/call_log_api.dart
    
    PROBLEM:
    Phone numbers used directly without validation:
    - Can contain invalid characters
    - No format checking (digits only)
    - No length validation
    
    IMPACT:
    - Native call function can crash with invalid format
    - Corrupted log entries with malformed numbers
    - Silent failures in logging API
    
    FIX: Add validation function:
        String? validatePhoneNumber(String? phone) {
          if (phone == null || phone.isEmpty) return null;
          final cleaned = phone.replaceAll(RegExp(r'[^0-9+]'), '');
          if (cleaned.length < 10) return null;
          return cleaned;
        }
    
    SEVERITY: HIGH (Prevents crashes)
    EFFORT: 10 minutes

---

⚠️ CRITICAL BUG #3: Race Condition in Duplicate Call Detection
    Location: lib/main.dart, _isDuplicateLeadCall() method, line ~440
    
    PROBLEM:
    Method is not thread-safe. Multiple notifications can be processed
    concurrently, causing race conditions:
    
        bool _isDuplicateLeadCall(Map<String, dynamic> data) {
          final key = '${data["docname"]}_${data["mobile_no"]}_${data["customer_name"]}';
          if (_lastLeadCallKey == key && isRecent()) {
            return true;
          }
          _lastLeadCallKey = key;  // ← Not atomic! Race condition here
          _lastLeadCallAt = now;
          return false;
        }
    
    IMPACT:
    - Duplicate calls might not be detected
    - Or incorrectly marked as duplicates
    - Inconsistent behavior under high call volume
    
    FIX: Use Set-based tracking with timestamps:
        Set<String> _recentCalls = {};
        
        bool _isDuplicateLeadCall(String key) {
          if (_recentCalls.contains(key)) return true;
          _recentCalls.add(key);
          Future.delayed(Duration(seconds: 3), 
            () => _recentCalls.remove(key)
          );
          return false;
        }
    
    SEVERITY: HIGH (Data integrity)
    EFFORT: 15 minutes

---

⚠️ CRITICAL BUG #4: No Network Timeout or Retry Logic
    Location: All API files (login_api.dart, opportunities_api.dart, 
              call_log_api.dart, device_api.dart)
    
    PROBLEM:
    HTTP requests have NO timeout:
    
        final response = await http.get(uri);  // Can hang forever!
        final response = await http.post(uri, body: data);  // Same issue
    
    IMPACT:
    - App can freeze indefinitely on slow network
    - No automatic retry on temporary failures
    - Battery drain from hanging connections
    - Users think app crashed
    
    FIX: Add timeout to all http calls:
        final response = await http.get(uri)
          .timeout(Duration(seconds: 10), 
            onTimeout: () => throw TimeoutException('Request timeout'));
    
    Then add retry logic in main screens for critical calls.
    
    SEVERITY: HIGH (Prevents app hangs)
    EFFORT: 20 minutes

---

⚠️ CRITICAL BUG #5: Timer Not Cancelled on Error
    Location: lib/main.dart, startCountdown() method, line ~1179
    
    PROBLEM:
    Timer is created but not cancelled if error occurs during countdown:
    
        startCountdown() {
          Timer? timer = Timer.periodic(Duration(seconds: 1), (t) {
            try {
              // ... countdown logic
              makeCall();  // ← If this throws, timer not cancelled
            } catch(e) {
              // Error handler doesn't cancel timer!
              print("Error: $e");
            }
          });
        }
    
    IMPACT:
    - Memory leak (timer keeps firing)
    - Repeated setState() calls causing jank
    - Multiple makeCall() attempts
    - Potential null pointer exceptions
    
    FIX: Add proper error handling:
        try {
          makeCall();
        } catch(e) {
          timer?.cancel();  // ← Cancel timer
          rethrow;
        }
    
    SEVERITY: HIGH (Memory leak)
    EFFORT: 10 minutes

SECTION 5: PERFORMANCE & ARCHITECTURE CONCERNS
================================================================================

🔧 STATE MANAGEMENT ISSUE
    Current: Using setState() in large HomePage widget
    Impact: Entire widget rebuilds on any state change
    Performance: Slow with large opportunity lists
    Solution: Consider Provider, Riverpod, or BLoC pattern

🔧 API CALL INEFFICIENCIES
    Current: getOpportunities() only called on init
    Issue: Stale data if backend list changes
    Solution: Add pull-to-refresh or periodic refresh timer

🔧 MEMORY LEAK POTENTIAL
    Files: lib/main.dart
    Issues:
    - _notificationSub not cancelled if exception before dispose
    - _tokenRefreshSub same issue
    - _pauseTimer same issue
    Solution: Use try-finally or LifecycleProvider pattern

🔧 CALL STATE TRACKING ISSUE
    Current: App lifecycle detection assumes call completion after pause
    Issue: If user manually hangs up within app, completion dialog won't show
    Solution: Rely more on EventChannel, less on lifecycle events

🔧 NO DATABASE CACHING
    Current: All data fetched fresh from API every time
    Impact: No offline support, increased network traffic
    Solution: Cache opportunities and call history locally with HiveDB

🔧 FIREBASE INITIALIZATION
    Current: Firebase.initializeApp() called in main() and background handler
    Issue: Redundant initialization
    Solution: Initialize once, pass instance to handlers

🔧 PERMISSION HANDLING
    Current: Permissions requested during call initiation
    Issue: Late permission request causes UX hiccup
    Solution: Request permissions on app startup

🔧 MISSING CRASH REPORTING
    Gap: No Firebase Crashlytics or Analytics integration
    Impact: Can't track production issues
    Solution: Integrate Firebase Crashlytics and Analytics
